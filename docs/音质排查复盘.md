# 阶段4 音质问题排查复盘

> 从"门岗声音稀烂"到最终修复，完整踩坑记录。答辩/讲解用。
> 一句话结论：**根因是 BlackHole 虚拟声卡的设备名义采样率卡在 48k，被电话/微信的 8k 窄带读端做 48k→8k 转换时损坏；修复 = `TTS_SAMPLE_RATE=16000` ＋ 把 BlackHole 2ch 设备名义率设成 16000。中途因为一个有漏洞的自测，误判成"微信 DSP 不可解"并一度放弃。**

---

## 背景：问题怎么暴露的

阶段 4 把来电源从本地麦克风换成"微信语音 / 真电话经 BlackHole 虚拟声卡进管线"。功能上**端到端全程跑通**了——STT、对话、公司名单匹配、复述确认、入库、推企微、优雅收尾全部在线。但来访者反馈两个问题：

1. **门岗（TTS）的声音稀烂/串道/全损**（核心问题，本复盘主线）。
2. 说完一句到听到回话约 2s 延迟（次要，后查明大部分是微信/电话 VoIP 固有的网络腿 + 轮次检测，非本系统问题，未深究）。

关键的非对称现象（一开始就观察到、但没用对）：**访客上行（人声→我们 STT）完美，只有下行（我们 TTS→访客）烂**。

---

## 踩坑时间线（6 个坑）

### 坑 1：以为是"输出 24k 跑在 48k 声卡上"的简单错配
- **假设**：所有设备名义率 48k，但输出管线跑 24k，两者打架。
- **动作**：发现 CosyVoice 原生支持 48k → 把 `TTS_SAMPLE_RATE` 从 24000 改 48000，让输出全程 48k、零重采样。开场白字节数 194980→389960 翻倍，确认真出 48k。
- **结果**：❌ 还是烂。
- **当时的错**：只对齐了"我们写"和"设备名义率"，没考虑"读端（微信/电话）用什么率读"。

### 坑 2：写了个环回自测，得出"我方链路完美"——但测试本身有漏洞
- **假设**：BlackHole 16ch 是 16 声道，被当单声道麦克风读时串道。
- **动作**：写环回测试——往设备播一段真实语音、同时从**同一设备**录回，算归一化互相关。
- **结果**：2ch 和 16ch **都 1.000 完美无损**。据此判定"我们的管线和声卡链路干净，问题在外部"。
- **当时的错（致命）**：这个自测**同一进程、读写都用 48k**，强制设备跑 48k、两端对齐，所以必然 1.000。它**根本没复现真实场景**——真实是"Pipecat 进程按某率写 + 微信/电话另一个进程按 8k 窄带读 + 设备名义率卡 48k"。**一个测了等于没测、却给了我虚假信心的自测，是后面误判的直接原因。**

### 坑 3：对调两个虚拟设备的角色，仍没用
- **假设**：还是 16ch 当麦克风非标准，让微信用标准的 2ch 当麦克风。
- **动作**：重建多输出设备、对调系统输入、改 `.env`，让 微信麦克风=2ch、访客声音=16ch、Pipecat 输出→2ch。
- **结果**：❌ 还是烂。

### 坑 4（最大的坑）：误判为"微信麦克风 DSP，不可解"，并放弃
- **推理**：环回证明我方链路干净（坑 2 的假象）；换设备、换采样率四种组合都烂；访客上行 STT 完美 → 推断"毁音质的是微信语音通话对麦克风输入的 DSP（降噪/AGC/AEC/窄带编码），微信内部、关不掉"。
- **动作**：写进文档当**已知局限**，提交"阶段 4 通路达成/音质受限"，准备收工。
- **当时的错**：这是个**可解问题被判了死刑**。推理链看着自洽，但地基（坑 2 的环回结论）是错的。教训：当多个"证据"都指向同一个不可控外因时，要回头质疑自己的测试有没有真正复现现场。

### 坑 5：换 Continuity 真电话，仍烂 → 反而加固了错误结论
- **动作**：开 iPhone Continuity，用真实蜂窝电话拨入。
- **结果**：❌ 还是烂（"稍微好一点点"）。
- **当时的错**：真电话也烂，被我当成"接收侧 DSP 是普遍限制"的佐证。**其实真相相反**——真电话也是 8k 窄带读端，撞的是同一个 48k→8k 坏转换，只是再次指向同一个真正的根因，而我没认出来。

### 坑 6：被"采样率"点醒后，只改代码写入率，仍没用（差最后一块拼图）
- **触发**：拿给做音频的人听，一句"这明显是采样率不匹配了"，重开采样率方向。
- **动作 A**：意识到坑 2 的漏洞，重做**跨率测试**（按 48k 写、分别按 48k/16k/8k 读）：

  | 写 → 读 | 互相关 | 判定 |
  |---|---|---|
  | 48k → 48k | 1.000 | ✅ |
  | 48k → 16k | 0.983 | ✅ |
  | **48k → 8k** | **0.108** | ❌ 损坏 |

  **首次复现了故障**：当读端用 8k（电话/微信窄带）读、而链路是 48k 时，信号损坏。
- **动作 B**：固定 8k 读端，扫写入率找干净的：

  | 写 → 8k 读 | 互相关 | 判定 |
  |---|---|---|
  | 8000 | 0.118 | ❌ |
  | **16000** | **0.964** | ✅ |
  | 24000 | 0.115 | ❌ |
  | 48000 | 0.052 | ❌ |

  只有 **16k** 能干净喂 8k 读端。于是 `TTS_SAMPLE_RATE=16000`。
- **结果**：❌ 真机**还是烂**！
- **当时的错**：以为改"我们写多少"就行。但忽略了——**BlackHole 是哑环回，永远按"设备名义率"跑**。设备名义率仍卡在 48k，所以我们写的 16k 先被升采到 48k 存进设备，微信再从 48k 降到 8k → 撞回那个坏的 48k→8k。写入率不是真正的杠杆。

## 真正的根因 + 最终修复

### 根因（一图看懂）
```
                          BlackHole 2ch（设备名义率 = 48k ← 病根）
Pipecat 写 16k ──SRC升→ [ 48k 缓冲 ] ──SRC降到 8k──→ 微信/电话按 8k 读
                                          ↑ 这步 48k→8k 在 BlackHole 上是坏的（实测 0.108）
```
- BlackHole 是**哑环回设备**：写进 output 侧的样本，按**设备当前名义率**原样流到 input 侧；客户端用别的率打开时由 CoreAudio 做每客户端 SRC。
- 设备名义率卡在 48k 时，不管我们写 16/24/48k，都先被转成 48k；电话/微信麦克风按 8k 读，触发 **48k→8k** 这步坏转换 → 稀烂。
- 为什么上行没事：上行是 **Pipecat 用自己高质量重采样去读**（48k→16k，实测 0.983 干净），坏的是"BlackHole/CoreAudio 把 48k 降到 8k 喂给外部读端"这个特定方向。

### 修复（两件套，缺一不可）
1. **代码**：`TTS_SAMPLE_RATE=16000`（CosyVoice 原生出 16k）。
2. **设备级（不在代码里）**：音频 MIDI 设置 → 选 **BlackHole 2ch** → **输入和输出两侧的"格式/采样率都改成 16000 Hz**。

修完链路：Pipecat 写 16k = 设备名义率 16k（匹配、不转换）→ 微信/电话读 8k = 干净的 **16k→8k 整数(2:1)降采样**。实测**音质完全正常**。

> ⚠️ 第 2 件是 Audio MIDI 设置里的**设备级配置、不进仓库**。换机器、重置音频、系统更新后都要复查。`scripts/audio-on.sh` 已加提醒。CLAUDE.md 其实早写过"通话音频常是 8k，选对应版本"——一开始就该往这想。

## 方法论教训（答辩可讲的点）

1. **自测必须复现真实现场的所有维度**。坑 2 的环回测试漏了三个关键维度：跨进程、读写率不一致、设备名义率不受控。一个"看起来通过"的测试，比没有测试更危险——它给了错误的信心，直接导致坑 4 的误判。
2. **别急着把问题归给不可控外因**。"微信 DSP 关不掉"听着合理，但它让一个可解问题被判死刑。证据链自洽 ≠ 地基正确；多条证据都指向同一外因时，优先回头质疑测试方法。
3. **找对真正的杠杆变量**。坑 6 证明"我们写多少"不是杠杆，"设备名义率"才是。对哑环回设备，关键永远是设备名义率与读端率的关系。
4. **非对称现象是金线索**。"上行好、下行烂"从第一天就在提示：问题不在编码/链路本身，而在某个方向特有的转换。早点深挖这条非对称，能少走很多弯路。
5. **外部视角值钱**。卡在错误结论里出不来时，一个做音频的人一句话就把方向掰回来了。
